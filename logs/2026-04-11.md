## 6. 实测期间遇到的重要问题与解决方案

### 问题 A：ROS 2 仿真时间死锁（Critical Time Sync Deadlock）

**症状：**
- Node 5 启动后，永远卡在日志 `Waiting for Nav2 to become active...`
- 尽管 Nav2 系统已在 RViz 中成功定位小车，但 `waitUntilNav2Active()` 始终无法返回
- 导致订阅器 `/vln_instruction` 无法创建（`Subscription count: 0`）

**根本原因分析：**
你现在写一下node5完成的md文件，你来写，你先写一下1. 用户在命令行用 `--ros-args -p use_sim_time:=True` 传参，但这个参数**仅给主节点**
2. `nav2_simple_commander.robot_navigator.BasicNavigator` 是一个**独立的内部 ROS 2 节点**，它不继承主节点的参数
3. 主节点活在**仿真时间**里，但 `BasicNavigator` 活在**真实物理时间**里
4. 两边时间戳差十万八千里，导致 `waitUntilNav2Active()` 的超时检查永远无法触发

**解决方案：**
- 在 `_on_init_timer_callback()` 中，在创建 `BasicNavigator()` 后立即强制设置其参数：
  ```python
  use_sim_time_param = Parameter('use_sim_time', Parameter.Type.BOOL, True)
  self.navigator.set_parameters([use_sim_time_param])
  ```
- 关键还有延迟初始化：`waitUntilNav2Active()` 原本在 `__init__` 中同步阻塞，此时 `rclpy.spin()` 还未启动，无法处理 `/clock` 话题消息，造成事实上的等待超时。改为在定时器回调中初始化，此时 `spin()` 已启动，`/clock` 消息可被正确处理。

**修改文件：**
- [src/vln_nav2_bridge/vln_nav2_bridge/vln_node_local.py](src/vln_nav2_bridge/vln_nav2_bridge/vln_node_local.py#L97-L101)

### 问题 B：参数声明碰撞（ParameterAlreadyDeclaredException）

**症状：**
```
rclpy.exceptions.ParameterAlreadyDeclaredException: ('Parameter(s) already declared', ['use_sim_time'])
```

**根本原因：**
- ROS 2 框架自动为所有节点声明 `use_sim_time` 参数
- 代码中再调用一次 `declare_parameter("use_sim_time")` 就冲突了

**解决方案：**
- 删除冗余声明，只在 `__init__` 中读参数（ROS 2 自动提供）：
  ```python
  try:
      use_sim_time = self.get_parameter("use_sim_time").value
  except Exception:
      use_sim_time = False
  ```

- [src/vln_nav2_bridge/vln_nav2_bridge/vln_node_local.py](src/vln_nav2_bridge/vln_nav2_bridge/vln_node_local.py#L21-L33)

### 问题 C：模型加载版本兼容性（Params4bit Constructor Mismatch）

**症状：**
```
Params4bit.__new__() got an unexpected keyword argument '_is_hf_initialized'
```

**根本原因：**
- 新版 `transformers` 库在构造 `BitsAndBytesConfig` 时，会向 `Params4bit` 传入新参数 `_is_hf_initialized`
- 用户环境的 `bitsandbytes` 版本较旧，不认识这个参数，导致加载失败

**解决方案：**
- 在模型加载前，monkey-patch `Params4bit.__new__` 以自动弹除不认识的参数：
  ```python
  def _patch_params4bit_constructor():
      original_new = Params4bit.__new__
      def _patched_new(cls, *args, **kwargs):
          kwargs.pop("_is_hf_initialized", None)
          return original_new(cls, *args, **kwargs)
      Params4bit.__new__ = staticmethod(_patched_new)
  ```

- [src/vln_nav2_bridge/vln_nav2_bridge/qwen_model_wrapper.py](src/vln_nav2_bridge/vln_nav2_bridge/qwen_model_wrapper.py#L6-L27)
- [src/vln_inference/run_inference_cli.py](src/vln_inference/run_inference_cli.py#L11-L31)

### 问题 D：Meta Tensor 运行时错误（GPU/CPU Offload Dispatch Bug）

**症状：**
```
Tensor.item() cannot be called on meta tensors
```

**经过复杂的诊断流程，发现：**
1. `device_map="auto"` + `llm_int8_enable_fp32_cpu_offload=True` 组合时，transformers 会把某些权重的占位符设为"meta tensors"
2. 调用 `.to("cuda")` 强制将输入张量移到 CUDA，但模型的某些内部层仍是 meta tensors，二者不匹配
3. 推理时访问 meta tensor 的值就会崩溃

**解决方案：**
采用"强制 CPU 路径"方案，彻底绕过 GPU offload 的复杂性：
- 新增 `--force-cpu` 命令行开关，使用纯 CPU 推理配置：
  ```python
  def _build_model_cpu(model_path: str):
      return Qwen3VLForConditionalGeneration.from_pretrained(
          model_path,
          device_map={"": "cpu"},
          torch_dtype=torch.float32,
          low_cpu_mem_usage=True,
          trust_remote_code=True,
      )
  ```
- wrapper 默认传入 `--force-cpu`
- 子进程环境变量强制 `CUDA_VISIBLE_DEVICES=-1`（GPU 对子进程不可见）

**修改文件：**
- [src/vln_inference/run_inference_cli.py](src/vln_inference/run_inference_cli.py#L50-L57)
- [src/vln_nav2_bridge/vln_nav2_bridge/qwen_model_wrapper.py](src/vln_nav2_bridge/vln_nav2_bridge/qwen_model_wrapper.py#L168-L173)

**权衡说明：**
- 优点：稳定性极高，完全避免了 meta tensor 和 offload 相关的坑
- 缺点：推理速度显著下降，从原来的 ~1.2s 延迟增加到 30s-1min（取决于内存带宽）
- 可接受性：对于离线实验和验证流程，这个延迟通常是可承受的；对于实时系统，后续可考虑单 GPU 推理或模型量化加速

### 问题 E：坐标空间不一致（Coordinate System Mismatch）

**症状：**
- 用户指令：`"Go to the right shelf with purple boxes"`
- 模型返回的 JSON：`{"x": 2.5, "y": 1.2, "yaw": 0.0}`
- 预期的真实坐标：`{"x": -6.78, "y": 10.96, "yaw": 0.0}`
- 结果：小车去了错误的位置

**根本原因：**
1. 原来的关键词规则表是针对旧地图坐标系（[-3, 4] 范围）编写
2. 当前实际仓库地图范围是 X: [-8, 10]，Y: [-12, 15]
3. 解析器优先相信模型输出的 JSON，导致偏差坐标覆盖了已知的正确目标

**解决方案：**
（两层并行修复）

1. **改变解析优先级**：访问指令中的已知关键词优先于信任模型输出的 JSON
   ```python
   def convert(self, instruction: str, model_output: str):
       # 第一优先：检查指令中是否包含已知目标名称
       parsed = self._fallback_from_instruction(instruction)
       method = "instruction_fallback"
       
       # 第二优先：解析模型返回的 JSON
       if parsed is None:
           parsed = self._parse_json_pose(model_output)
           method = "json"
       
       # 第三优先：通用关键词匹配
       if parsed is None:
           parsed = self._fallback_from_text(instruction, model_output)
           method = "fallback"
   ```

2. **更新关键词坐标表和安全范围**：
   - 新的安全范围：X: [-8.0, 10.0]，Y: [-12.0, 15.0]
   - 更新目标坐标映射：
     - `purple box/boxes/shelf/right shelf` → (-6.78, 10.96, 0.0)
     - `plant` → (-0.43, -2.92, 0.0)
     - `chair` → (-0.54, -0.69, 1.57)

3. **同步模型 prompt**：在子进程的 CLI 脚本中也加入地图坐标记忆，让模型推理时有参考：
   ```
   "Here is the exact map memory of the environment: "
   "[shelf with purple boxes is at x=-6.78, y=10.96], "
   ...
   ```
